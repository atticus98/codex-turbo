# 📋 全局共享规范

## 🎯 基本原则

1. 交流对话过程中，所有思考、分析、解释和回答必须使用简体中文。
2. 重要判断尽量附 `file#method:line`、命令、方法名、参数名、真实行为说明。
3. 不要空泛地说"应该""可能"，转化为具体可执行的操作建议。
4. **Skills 优先** —— 优先使用 Skills 驱动问题处理

---

## 📂 项目规范

1. 严格遵守当前工作目录下的 `AGENTS.md` 定义的项目规范。
2. 若本规范与 `AGENTS.md` 有冲突，以更高层级规范为准，同时尽量兼容项目规范。

---

## ✅ 完成标准

1. 回答关于配置、执行优先级、超时等待、工具行为的问题时，优先引用真实代码位置(`file#method:line`)。
2. 回答"为什么像卡住""为什么不等待了"这类问题时，先解释运行边界与默认值。
3. 如果声称"已完成""已修复""已通过"，**必须先有真实验证证据**。
4. 如果没有验证，就明确说"未验证"，不要暗示已经验证。


---

## 📊 质量标准

### 🏗️ 工程原则

**架构设计**：遵循 KISS（Keep It Simple）、YAGNI（You Aren't Gonna Need It）原则，不做过度设计。
> 避免过度抽象和学术化设计，优先编写简洁直观的代码。
> 当前不需要的功能不要提前实现；需要时再重构。
> 过早优化与过度设计均属反模式。

**代码质量**：
- 追求代码的可读性和易理解，不要过度抽象和封装。
- 核心流程必须有简体中文注释（关键流程、核心逻辑、重点难点）
- 删除无用代码，修改功能不保留旧的兼容性代码

**测试要求**
  - **测试驱动** —— 可测试设计，单元测试覆盖
  - **超时配置** —— 后台执行单元测试时，建议设置最大超时时间 60s（可按项目覆盖），避免任务卡死

## ⚠️ 危险操作确认机制

### 🚨 高风险操作清单

执行以下操作前**必须获得明确确认**：

- **文件系统** —— 删除文件/目录、批量修改、移动系统文件
- **系统配置** —— 修改环境变量、系统设置、权限变更
- **数据操作** —— 数据库删除、结构变更、批量更新
- **网络请求** —— 发送敏感数据、调用生产环境 API
- **包管理** —— 全局安装/卸载、更新核心依赖


### 📝 确认格式模板

```text
⚠️ 危险操作检测！

操作类型：[具体操作]
影响范围：[详细说明]
风险评估：[潜在后果]

请确认是否继续？[需要明确的 "是"、"确认"、"继续"]
```
